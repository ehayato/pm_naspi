「Raspberry Piを活用したハイブリッドセキュアストレージおよび自動運用Bot開発プロジェクト」

### プロジェクトマネージャー
申請者本人（単独実施）

### 承認者
ゼミ指導教員

---

## 2. プロジェクト発足の背景

1. 利用中のクラウドサービス（ProtonDrive）がiPadアプリ経由でのPDF書き込みに対応していない
2. iPad純正ファイル管理アプリではクラウド同期が遅延・不安定
3. 特定のデバイスやアプリに依存しない、高速で堅牢なファイル管理基盤が必要

---

## 3. プロジェクト目的

不完全なファイル管理システムに依存せず、以下を実現する：

1. iPadから**SMB経由で即座にアクセス可能**なローカルNAS環境を構築
2. **rclone bisync**によりローカルとクラウドストレージの**自動双方向同期**（30分毎）を実装
3. **Discord Bot**を通じてデバイス問わず手動同期・状態監視・操作を実現
4. **rsync**による日次・週次の差分バックアップで、最大2週間分のデータ保護を確立

---

## 4. 技術的意義

本プロジェクトを通じて、以下の実践的スキルを習得する：

1. Linux（Raspberry Pi）システム管理・Docker運用
2. Samba（SMB）によるネットワークストレージ設計と権限管理
3. rclone bisyncを用いたクラウドストレージとの双方向同期設計
4. Discord Bot（Python / discord.py）を用いたAPI駆動・非同期プログラミング
5. cronによるジョブスケジューリングとrsyncを用いたバックアップ設計
6. バージョン管理・ドキュメント作成を含むソフトウェア工学の実践

---

## 5. 最終成果物

### システム構成図

```
【通常時】
クラウドストレージ（ProtonDrive）
    ↕ rclone bisync（30分毎 + Bot手動）
/mnt/data/cloud（ローカルコピー）
    ↕ Samba（SMB）で公開
iPad → SMB接続 → 読み書き

【バックアップ時のみ】
/mnt/data/cloud（クラウドストレージのローカルコピー）
    ↓ rsync（日次・週次 cron + Bot手動）
HDD マウント → 差分バックアップ実行 → アンマウント
（クラウドストレージデータのローカル世代管理、最大2週間）
```

### 主成果物

**1: Samba NAS環境** ✅ 完成済み
- Raspberry Pi上のSambaサーバー（Docker運用）
- `Cloud-Sync` 共有：`/mnt/data/cloud`（rclone同期先と連携）
- `Local-NAS` 共有：`/mnt/data/local`（ローカル専用領域）
- iPadからSMB接続でファイルの読み書きが可能な状態

**2: Discord Bot基盤** ✅ 完成済み
- 起動通知・稼働確認機能（Docker運用）
- 以下のコマンド群を追加実装予定：
  - `/sync`：rclone bisync の手動即時実行
  - `/status`：ストレージ使用率・同期状態の確認
  - `/backup`：rsync バックアップの手動実行
  - `/log`：直近の同期・バックアップログ確認

**3: rclone bisync（クラウドストレージ連携）** 🔄 移植・設定待ち
- Arch Linux環境での動作実績あり
- Raspberry Piへの移植とcron設定（30分毎）
- 競合発生時のハンドリング設定
- 初回 `--resync` 実行手順のドキュメント化

**4: rsync 差分バックアップ** 📋 未着手
- バックアップ対象：`/mnt/data/cloud`（クラウドストレージのローカルコピー）
- 日次：`/mnt/data/cloud` → `/mnt/data/backup/daily/`
- 週次：`/mnt/data/cloud` → `/mnt/data/backup/weekly/`
- 保持期間：最大2週間（世代管理スクリプト）
- HDD運用：**普段はアンマウント**、バックアップ時のみマウント→実行→アンマウント
- cronおよびBot（`/backup`コマンド）から実行可能

### 副成果物
- 設計ドキュメント・セットアップガイド
- テスト結果レポート
- GitHubリポジトリ（ソースコード・ドキュメント公開）

## 10. 制約条件

### 技術的制約
- Raspberry Pi 4のリソース制限（CPU・RAM）
- クラウドストレージのAPI制限により、同期頻度に上限あり
- rclone bisyncの競合リスク低減のため、同期間隔は30分毎に設定
- ローカルネットワーク内限定のSMBアクセス（外部公開なし）

### 時間的制約
- ゼミの他の課題・講義との並行実施
- 単独開発のため、作業が集中するリスクあり

### リソース制約
- メンバーは一人
- Raspberry Pi・HDD・iPad・ネットワーク機器は既有資産
- 外部ライブラリ・OSSの活用（ライセンス確認必須）

## 12. 主要なリスク（高レベル）

| リスク | 影響度 | 発生確率 | 対応策 |
| - | - | - | - |
| Raspberry PiのCPUスペック不足によるSMB転送速度の低下 | 高 | 高（既知） | 暗号化無効化で改善確認済み；根本解決はPi5へのアップグレード |
| rclone bisyncとクラウドストレージの競合・不具合 | 高 | 中 | Arch環境での事前検証済み；競合ポリシーを明示的に設定し、初回`--resync`手順をドキュメント化 |
| 同期・バックアップの同時実行による負荷集中 | 中 | 中 | cronジョブの実行時間を分散；軽量な設定を優先 |
| Discord API / rclone仕様変更 | 中 | 低 | 公式ドキュメント継続監視；定期テスト実施 |
| 単独開発による作業遅延 | 高 | 中 | MVPを明確化；①②を完成済みとしてスケジュールに反映済み |
| HDDの突然の障害 | 高 | 低 | 普段はアンマウント運用で通電時間を最小化；正副2系統（クラウド＋HDD）でデータ保護 |
