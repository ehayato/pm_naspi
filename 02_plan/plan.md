## 1. 文書の目的

本計画書は、[プロジェクト憲章](../01_init/project-charter.md)で定義された目的・制約・成果物をもとに、PMBOK第6版の考え方に沿って実施計画を具体化するものである。対象は、Raspberry Piを活用したハイブリッドセキュアストレージおよび自動運用Bot開発プロジェクト全体とする。

---

## 2. プロジェクトの前提と方針

### 2.1 前提

- Raspberry Pi 4 (4GB)と外付けHDDは既に利用可能である
- Samba NAS基盤とDiscord Bot基盤は完成済みである
- rclone bisyncの移植とresticバックアップは未完了である
- 開発は単独実施であり、意思決定はプロジェクトマネージャーが行う

### 2.2 実施方針

- 既存資産を最大限活用し、追加調達は最小化する
- まずは動作する最小構成を確立し、その後に自動化・安定化を進める
- 失敗時の復旧手順を先に整備し、破壊的な運用を避ける
- 仕様変更は記録し、憲章の目的に反しない範囲で管理する

---

## 3. プロジェクト目標と成功基準

### 3.1 目標

1. iPadからSMB経由でCloud-Sync・Local-NASへ安定して読み書きできる環境を維持する
2. rclone bisyncにより、クラウドストレージとローカルコピーの双方向同期を30分毎に自動実行する
3. Discord Botから同期・状態確認・バックアップ操作を実行できるようにする
4. resticによる日次(7日間)・週次(2週間)の差分バックアップを行い、3週間分の世代管理を実現する
5. 設計・運用・復旧を含むドキュメントを整備し、ゼミ発表可能な状態にする

### 3.2 成功基準

- SMB接続で読み書きが安定し、主要端末で操作可能である
- rclone bisyncが競合なく継続動作し、同期ログが追跡可能である
- Discord Botの全コマンドが期待どおり応答する
- バックアップが所定の周期で実行され、復元可能性が確認される
- 1週間の連続稼働テストで重大障害が発生しない
- ドキュメントが第三者にも再現可能な粒度で整備されている

## 4. スコープ管理計画

### 4.1 スコープ内

- Raspberry Pi上のDocker運用
- Samba設定、認証、権限管理
- rclone bisyncの設定、自動化、競合対策
- Discord Botの機能拡張
- resticバックアップ設計、自動化、世代管理
- cronによるジョブスケジューリング
- 運用手順書、セットアップガイド、テスト記録の作成

### 4.2 スコープ外

- 新規のWebフロントエンド開発
- 外部公開を前提としたサービス運用
- ハードウェアの新規大量調達
- クラウドストレージサービス自体の変更や契約追加

### 4.3 スコープ確定方法

- 追加要件は憲章の目的との整合を確認する
- 既存機能の置き換えではなく、目的達成に必要な最小機能を優先する
- 変更が生じた場合は、影響範囲・工数・リスクを簡潔に記録する

## 6. スケジュール管理計画

### 6.1 マイルストーン

| フェーズ | 期間 | 完了条件 |
| - | - | - |
| 立ち上げ | 完了 | Samba・Bot基盤の初期実装完了 |
| 詳細設計 | 1週間 | 同期・バックアップ・通知の設計確定 |
| rclone移植 | 2週間 | Raspberry Pi上でbisyncが安定動作 |
| バックアップ実装 | 2週間 | resticと世代管理が自動化 |
| Bot拡張 | 2週間 | `/sync` `/status` `/backup` `/log` 実装 |
| テスト・安定化 | 2週間 | 連続稼働試験と不具合修正完了 |
| 文書化・発表準備 | 1週間 | ガイドと発表資料が完成 |

### 6.2 スケジュール管理方針

- 重要ジョブは相互干渉しない時間帯に配置する
- cron設定は段階的に適用し、手動検証後に自動化へ移行する
- 週次で進捗を確認し、遅延があればスコープを調整する

## 7. コスト管理計画

### 7.1 予算方針

- 基本方針は既有資産の活用であり、原則として追加費用を発生させない
- ソフトウェアはオープンソース中心で構成する
- Pi 5へのアップグレードは必要性が明確な場合のみ検討する

### 7.2 管理方法

- 追加費用が発生する場合は、目的・代替案・効果を比較して記録する
- ハードウェア費、消耗品費、追加サービス費を分けて管理する

---

## 8. 品質管理計画

### 8.1 品質方針

- 動作することだけでなく、再現できることを品質基準に含める
- 本番相当の設定変更は、検証環境または段階的適用で確認する
- エラー時のログが読めることを最低条件とする

### 8.2 品質指標

- Sambaの読み書きが安定していること
- bisyncが意図しない衝突を起こしていないこと
- バックアップ先が世代管理されていること
- Botコマンドが想定外の入力で破綻しないこと

### 8.3 品質保証活動

- 手動テスト結果の記録
- 代表的な障害シナリオの復旧確認
- 設定変更時の差分確認

---

## 9. リソース管理計画

### 9.1 使用資源

- Raspberry Pi 4または必要時のPi 5
- SSD 1TB
- HDD 4TB
- iPad、PC、ネットワーク機器
- Docker、Samba、rclone、restic、Discord Bot実行環境

### 9.2 管理方針

- HDDは通常時アンマウントし、バックアップ時のみマウントする
- CPU・RAM負荷が高い処理は実行時刻をずらす
- 作業時間を小刻みに区切り、単独開発の遅延を抑える

## 11. リスク管理計画

### 11.1 主要リスクと対応

| リスク | 影響 | 対応 |
| - | - | - |
| Raspberry Piの性能不足 | 高 | 負荷の高い処理を分散し、必要ならPi 5を検討する |
| rclone bisyncの競合や不整合 | 高 | 同期間隔を固定し、初回resync手順とログ確認を徹底する |
| バックアップと同期の同時実行 | 中 | cron時刻を分離し、実行順を固定する |
| HDD障害 | 高 | 通電時間を短くし、クラウド側とローカル側の二重保護を維持する |
| 単独開発による遅延 | 高 | MVPを優先し、後回し可能な改善を切り分ける |
| SSD・HDDの暗号化 | 高 | ストレージ盗難時にデータを保護するための対策。対策は検討中。 |

### 11.2 リスク監視

- 失敗ログの蓄積
- 週次の障害・遅延レビュー
- 設定変更時の影響確認

---

## 12. 調達管理計画

- 原則として新規調達は行わない
- 必要時のみ、Pi 5やネットワーク機器の追加を検討する
- 調達が必要な場合は、性能改善効果と代替案を比較する

---

## 13. ステークホルダー管理計画

- 申請者が実行・判断・記録を一元管理する
- 指導教員には節目ごとに成果と課題を報告する
- 発表前に、技術説明と成果の見せ方を調整する

---

## 14. 変更管理計画

- 要件追加は、目的との整合、工数、リスクを確認して判断する
- 実装済み機能への影響がある場合は、変更内容を記録する
- 変更は原則として小さく分割し、検証後に反映する

---

## 15. 構成管理計画

- 設定ファイル、スクリプト、文書をGitで管理する
- 重要設定は差分が追える形式で保存する
- 初期化手順と復旧手順は最新版を明示する

---

## 16. 実行管理と監視

- 定期ジョブの実行結果をログで確認する
- エラー検知時は手動同期・手動バックアップを優先する
- 連続稼働テスト期間中は、例外・遅延・失敗回数を記録する
